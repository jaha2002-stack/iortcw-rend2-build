# Rend2 Kit D: shader-side cubemap depth compare debug

This kit is designed to stop guessing.

Previous proofs:
- Kit A: torch dynamic light reaches renderer.
- Kit B: cubemap faces 0..5 render.
- Kit C: lightView=1 and faces 0..5 render from the light origin.

Key issue found:
The stock dlight fragment shader did not read `u_ShadowMap`, so a generated cubemap could not affect visible dlight shadows.

Kit D adds:
- dlight vertex shader exports world position.
- dlight fragment shader reads samplerCube `u_ShadowMap`.
- shader diagnostic modes 0..7.
- actual non-linear depth compare mode.

Run this exact sequence:

1. /exec rend2_kit_d_1_force_shadow.cfg
Expected: selected torch dynamic light becomes much dimmer. If not, shader/uniform path is not active.

2. /exec rend2_kit_d_7_direction_pattern.cfg
Expected: visible directional pattern/split. If mode 1 worked but this does not, world position or light origin is wrong.

3. /exec rend2_kit_d_3_cubemap_sample_debug.cfg
Expected: grayscale variation. If modes 1 and 7 work but this is flat, cubemap is not readable/empty/wrong texture state.

4. /exec rend2_kit_d_4_actual_nonlin_compare.cfg
Expected: possible hard torch shadows. If wrong, tune bias or test modes 5 and 6.

5. /exec rend2_kit_d_5_actual_linear_compare.cfg
6. /exec rend2_kit_d_6_inverted_nonlin_compare.cfg

Then:
/condump kit_d_result.txt
